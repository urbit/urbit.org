# Kelvin 409k to 408k Migration Notes

Use this reference as a starting checklist for desks moving from 409k-compatible source to 408k-compatible target. Verify current release notes and source before applying a rule mechanically.

## Common audit areas

- **Eager mark builds:** every mark under `/mar` must compile, including old or rarely used marks.
- **Date strings:** if generated paths or wires include `@da`, use the formatter recommended by the target release notes.
- **Persisted state:** if saved Gall state stores Hoon `type`, `vase`, dependency-defined structures, wrapper-saved nested `vase`s, or opaque imported values, add a state version and use release-provided migration helpers when required.
- **Strict library changes:** check changed standard-library gates and crypto APIs against the target release.
- **Generic helper composition:** if the compiler or runtime reports `have *` where a concrete key/path mold is needed, inspect point-free helper composition and replace it with an explicit gate naming the sample and product type.
- **External dependencies:** inspect imported desks and libraries with the same rigor as local source. A desk can advertise the target Kelvin while still importing incompatible older source.

## Patch rules

- Add `408` to `sys.kelvin` while preserving the repository's ordering style.
- Keep docket/app version bumps until after compatibility tests pass.
- Align dependency pins with commits tested on the target Kelvin.
- Document any temporary vendored override and decide whether it will be upstreamed, re-pinned, or removed before release.

## Minimum validation

- Fresh fake-ship install on the target Kelvin.
- Source-Kelvin to target-Kelvin upgrade with non-empty state.
- All marks, agents, generators, and tests compile.
- Primary pokes, watches, scries, HTTP routes, and cross-desk flows succeed.
